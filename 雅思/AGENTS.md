# IELTS 学习资料与网页维护指南

本仓库既是 IELTS 听、说、读、写学习资料库，也是一个逐步建设中的个人学习网站。后续代理在任何电脑上修改本仓库时，都应先阅读本文件；进入含有更深层 `AGENTS.md` 的目录后，还要同时遵守该目录的细化规则。若两者冲突，以距离目标文件最近的 `AGENTS.md` 为准。

## 总体目标

- 通过网页统一访问 `listening`、`speaking`、`reading`、`writing` 四类资料。
- 使用共享 CSS 保持页面风格一致，让排版调整可以一次生效于所有页面。
- 优先建设写作模块，并完整保留作文的多个版本、ChatGPT 批改意见和修改轨迹。
- 保留 Markdown 作为学习内容的主要来源，网页应读取或构建这些内容，不要把同一正文手工复制到多个 HTML 文件中。
- 网站应能在不同电脑上从仓库重新构建和运行，不依赖某台电脑的绝对路径、私人配置或未提交数据。

## 当前目录角色

- `listening/`：听力题目、原文、复盘及听力专项资料。
- `speaking/`：口语题目、回答和练习记录。
- `reading/`：阅读资料、词汇辨析、图片和复盘记录。
- `writing/`：写作真题、题型分类、作文版本和批改历史；当前优先开发。
- 根目录 Markdown：跨科目的计划、复盘和综合资料。

## 不可突破的目录边界

- `listening/`、`speaking/`、`reading/`、`writing/` 是只读学习资料目录；网页开发不得在其中新增、修改、移动、重命名或删除任何文件。
- 即使为了适配网页，也不得整理、迁移或改写这四个目录中的内容。
- 网站可以读取或链接这些资料，但所有索引、转换结果、缓存、批改记录副本和展示用数据都必须保存在网页自己的目录中。
- 若用户提出的网页功能必须改变四个目录，停止实施并先向用户说明；不得自行放宽此规则。
- 四个目录内已有的 `AGENTS.md` 只在用户明确要求处理相应学习资料时生效；纯网页任务不得以这些文件为理由修改资料目录。

## 网站建设约定

- 网页源文件仅放在根目录的网页入口文件及 `assets/` 等网页专用目录中。
- 网站应有统一入口，并提供听、说、读、写四个清晰导航项。
- 页面布局、字体、颜色、间距、内容宽度、表格、代码块和响应式规则集中放在共享样式中，避免在内容页重复内联 CSS。
- 优先使用可复用的页面布局和组件；相同功能不得为每篇资料单独实现一套。
- 内容层与展示层分离：学习内容保存在 Markdown 或结构化数据中，网页负责导航、渲染、筛选和版本展示。
- 所有本地资源使用仓库内相对路径；文件名含中文时也必须正确处理链接和 URL 编码。
- 桌面端和手机端都应可读；不得依赖固定屏幕宽度。
- 加入新的前端框架、数据库、云服务或大型依赖前，先说明必要性。优先选择安装简单、可本地运行、可静态构建的方案。
- 除非用户明确要求，不要把未完成的草稿、私人批注或学习记录上传到外部服务。

如果仓库尚未确定具体前端技术，不要擅自锁定长期架构。可以先实现最小可用写作页面，但必须让后续听、说、读模块能够复用同一套导航、布局和样式。

## 写作版本与批改历史

新增作文时，应把“题目资料”和“个人作文历史”区分开。已有真题文件继续作为题目来源；个人作文及批改记录放在对应题型和题目附近的 `essays/` 目录中，避免修改真题原文来记录作文。

每个作文版本使用一个独立 Markdown 文件，直接放在对应题型的 `essays/` 目录中。不要在同一文件中累积或覆盖多个版本：

```text
writing/<题型>/essays/
  C{册号}-Test{测试号}-Task{任务号}-Essay-V01.md
  C{册号}-Test{测试号}-Task{任务号}-Essay-V02.md
  C{册号}-Test{测试号}-Task{任务号}-Essay-V03.md
```

模板文件固定为 `writing/TEMPLATE-CNN-TestNN-TaskN-Essay-VNN.md`，其中 `N` 表示需要替换的数字。正式文件名使用 `C19-Test1-Task1-Essay-V01.md` 这类格式。

版本号固定使用两位数字 `V01`、`V02`、`V03`，方便文件排序。每个版本文件保存该版本的完整作文、个人思考、ChatGPT 点评和修改计划。网页通过题型 `essays/` 目录下与题号匹配的 `C*-Test*-Task*-Essay-VNN.md` 文件数量识别历史版本数，并按文件名中的版本号排序。

版本文件不使用 front matter，也不重复写题号或版本标题；题号和版本号均从目录名与文件名获得。文件使用以下稳定结构：

```markdown
## Essay

（本版作文原文）

## My thoughts

（本版写作前后的个人思考）

## ChatGPT feedback

（ChatGPT 对本版作文的评分、问题和修改建议）

## Revision notes

（本版总结）
```

维护历史时遵守以下规则：

- 每次重要修改新增一个版本文件，不覆盖、重命名或编辑旧版本文件；旧版本需要补充点评时除外。
- 每个版本保留完整作文，确保脱离聊天记录后仍可阅读和比较。
- ChatGPT feedback 必须保存在对应版本文件中，不把多轮建议混在一起。
- 保留用户原文；纠错稿、范文或 ChatGPT 重写稿必须明确标注，不冒充用户版本。
- 日期统一使用 `YYYY-MM-DD`；同一天有多版时仍按 `V01`、`V02`、`V03` 递增编号。
- 网页至少应支持按版本查看；以后可增加并排比较、修改高亮、评分趋势和问题标签。
- 若加入评分，分别记录 Task Achievement/Task Response、Coherence and Cohesion、Lexical Resource、Grammatical Range and Accuracy，并注明是估分而非官方成绩。

## 内容和编码规则

- 文本文件统一使用 UTF-8，修改后检查中文、英文撇号和特殊符号是否出现乱码。
- 不无故改变 IELTS 题目原文、图片、引用来源或既有答案。
- Markdown 图片使用相对路径，不使用容易失效的本机绝对路径。
- 不提交 Cookie、密码、API Key、登录会话、个人令牌或浏览器配置。
- 生成内容时尽量保留清晰的标题层级，便于网页建立目录、搜索和索引。

## 修改流程

1. 先阅读根目录及目标目录中适用的全部 `AGENTS.md`。
2. 检查现有结构和 Git 改动，只修改本次任务涉及的文件，不覆盖用户的其他未提交内容。
3. 尽量沿用已有命名、目录和内容格式；需要新增约定时同步更新相关说明。
4. 修改网站后，至少验证目标页面能打开、导航和资源链接有效、Markdown 正常渲染、手机宽度下内容不横向溢出。
5. 修改作文历史后，检查版本编号、日期、Essay 与 Feedback 的对应关系，以及旧版本是否仍完整存在。
6. 交付时简要说明改了什么、如何查看，以及尚未解决的问题；除非用户明确要求，不自动提交或推送 Git。

## 优先级

当前开发顺序为：

1. 写作资料入口与题型/题目导航。
2. 作文多版本与 ChatGPT 批改历史展示。
3. 全站共享 CSS、响应式布局和统一导航。
4. 逐步接入听力、口语和阅读资料。
5. 搜索、筛选、版本比较、学习统计等增强功能。

在前两项可稳定使用之前，避免把时间投入到登录系统、复杂后端、动画或与学习无关的视觉效果上。
